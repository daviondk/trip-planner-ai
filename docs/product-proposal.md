
# Product Proposal: Персонализированный помощник для планирования путешествий

## 1. Обоснование идеи — прикладная задача

**Прикладная задача:** Автоматизировать и персонализировать процесс планирования путешествия, преобразовав его из многочасового ручного поиска и сопоставления данных в интерактивный диалог с ИИ-агентом.

**Текущая проблема:**
Процесс планирования поездки для современного путешественника фрагментирован и отнимает много времени.
1.  **Информационный шум:** Пользователь вынужден просматривать десятки сайтов (блоги, агрегаторы отелей, карты, форумы) для сбора информации о достопримечательностях, транспорте, стоимости и логистике.
2.  **Отсутствие персонализации:** Существующие сервисы (типа TripAdvisor или готовые гиды) предлагают стандартные, "заезженные" маршруты, которые редко учитывают уникальные интересы пользователя (например, "гастрономический тур по пивным" или "фото-прогулка по малоизвестным крышам").
3.  **Сложность адаптации:** Любое изменение в планах (смена отеля, плохая погода, продление поездки) требует частичного или полного перезапуска цикла планирования. Гибкость в реальном времени практически отсутствует.

**Почему агентный подход, а не просто сайт-агрегатор?**
Задача требует не просто поиска, а *синтеза* информации из разных источников с учетом динамического контекста. Агентная система позволяет:
*   **Вести диалог:** Уточнять предпочтения в процессе общения, а не через жесткую форму.
*   **Комбинировать знания:** Один агент ищет отели, второй — рестораны, третий — транспорт. Оркестратор собирает это в связанный маршрут.
*   **Управлять состоянием:** Помнить, что пользователь путешествует с детьми (и не предлагать ночные клубы) или что у него ограниченный бюджет на конкретный день.
*   **Адаптироваться на лету:** Перестроить маршрут, если пользователь сообщает "я устал и хочу сегодня просто погулять в парке, а не идти в музей".

## 2. Цель и метрики успеха

**Цель проекта:** Создать PoC мульти-агентной системы, которая по запросу пользователя (город, даты, интересы) генерирует персонализированный маршрут с привязкой к карте, подбирает опции бронирования и позволяет экспортировать результат.

### Метрики успеха

| Категория | Метрика | Целевое значение (PoC) | Способ измерения |
| :--- | :--- | :--- | :--- |
| **Продуктовые** | Время генерации первого черновика маршрута | ≤ 30 секунд | Замер времени от отправки запроса до вывода списка активностей. |
| | Завершенность сценария | Пользователь заканчивает планирование в 70% сессий | Логирование действий пользователя в интерфейсе. |
| | Релевантность рекомендаций | Средняя оценка пользователем ≥ 4/5 | Опрос после демо (n=5). |
| **Агентские** | Точность классификации интентов (роутер) | Accuracy ≥ 90% | Тестирование на синтетическом датасете запросов. |
| | Успешность инструментальных вызовов (Tool calls) | ≥ 85% успешных вызовов API (карты, отели) | Мониторинг выполнения tools в логах агента. |
| | Галлюцинации в описаниях мест | < 5% (описания должны соответствовать данным из API) | Выборочная экспертная проверка сгенерированных текстов. |
| **Технические** | p95 latency ответа агента (без учета бронирования) | ≤ 10 секунд | Мониторинг HTTP-запросов к API агента. |
| | Стоимость одной сессии планирования | ≤ $0.10 (на основе расхода токенов GPT-4o-mini) | Подсчет токенов через LangSmith. |
| | Доступность ключевых API (карты) | 99.9% (в рамках лимитов тестового аккаунта) | Мониторинг ошибок интеграций. |

## 3. Сценарии использования (включая edge-кейсы)

### Основные сценарии

**S1. Семейная поездка на выходные:**
Пользователь: "Планирую поездку в Суздаль на 3 дня с женой и ребенком (5 лет). Интересуемся историей и хотим попробовать местную кухню. Бюджет средний."
*   Агент-оркестратор определяет интент `plan_trip`.
*   Агент-планер (Planner Agent) запрашивает уточняющие вопросы (нужна ли машина, тип жилья).
*   Параллельно запускается поиск отелей (Booking Agent) и достопримечательностей (Info Agent).
*   Результат: Маршрут на карте: день 1 — заселение, обед в ресторане "Уездный", прогулка по Кремлю; день 2 — Музей деревянного зодчества + кафе с детской комнатой; вечер — дегустация медовухи (для взрослых). С опциями бронирования отеля.

### Edge-кейсы

| Сценарий | Ожидаемое поведение агента |
| :--- | :--- |
| **E1. Противоречивые предпочтения** | Пользователь: "Хочу недорого, но 5 звезд и чтобы был бассейн". Агент должен запросить уточнение приоритетов или показать варианты, где эти параметры пересекаются, с пометкой о компромиссе. |
| **E2. Недоступность API бронирования** | Сервис отелей временно не работает. Агент сообщает: "К сожалению, сейчас не могу показать цены, но вот список рекомендуемых отелей. Попробуйте позже". PoC не должен падать. |
| **E3. Изменение планов на ходу** | Пользователь: "Мы уже в городе, но устали. Отмени музей, предложи что-то спокойное рядом с отелем". Агент должен перезапустить поиск с новым контекстом (местоположение, состояние). |
| **E4. Слишком общий запрос** | Пользователь: "Хочу в Париж". Агент должен задать уточняющие вопросы: "На сколько дней? С детьми? Какие цели: музеи, шопинг или романтика?" |
| **E5. Prompt Injection** | Пользователь пытается внедрить в запрос инструкцию для смены роли агента. Системный промпт изолирован, а пользовательский ввод обрабатывается как данные, а не как инструкции (см. governance.md). |

## 4. Ограничения

### Технические
*   **Размер батча:** PoC работает в режиме реального времени для одного пользователя, без пакетной обработки.
*   **LLM:** Основная модель (на первом эиапе) — API YandexGPT.
*   **Интеграции:** Используются тестовые/публичные API с жесткими лимитами (Google Maps Places API (только чтение), парсинг публичных данных (как fallback), без реального бронирования).
*   **Хранилище:** Сессионное состояние (история диалога, выбранные варианты) хранится в оперативной памяти (in-memory).
*   **Инфраструктура:** Docker-контейнеры (FastAPI + Streamlit для UI) на сервере с 2 vCPU и 4GB RAM.

### Операционные
**Данные:** Используются публичные данные (Wikipedia, OpenStreetMap) и синтетические данные о достопримечательностях для нетестовых регионов.

## 5. Архитектурный набросок

**Модули и интеграции:**

1.  **Web Interface (Frontend):** Принимает запросы, отображает карту (через iframe), кнопки для выбора действий.
2.  **Orchestrator Agent (LangGraph):** Управляет состоянием диалога и потоком данных между специализированными агентами. Хранит `user_state` (бюджет, состав группы, текущий город).
3.  **Специализированные Агенты (Tools):**
    *   **Planner Agent:** Разбивает поездку на дни, определяет логику маршрута.
    *   **Info Agent (RAG):** Ищет информацию о достопримечательностях (использует Wikipedia API или векторную БД с описаниями).
    *   **Booking Agent:** Взаимодействует с API для поиска отелей или парсит сайты-агрегаторы.
    *   **Mapper Agent:** Формирует запросы к Google Maps API для отображения точек на карте и построения треков.
4.  **Memory (State):** Хранит контекст сессии.
5.  **Vector DB (ChromaDB):** Хранит эмбеддинги описаний мест для быстрого поиска по интересам.


## 6. Data Flow и делегирование LLM/Agent

| Шаг | Делегируется LLM/Агенту | Не делегируется (детерминированный код) |
| :--- | :--- | :--- |
| **Понимание запроса** | Извлечение сущностей (город, даты, интересы, бюджет) из свободного текста. Определение интента (новый план / изменение / вопрос). | Валидация формата дат, проверка наличия обязательных полей. |
| **Формирование плана** | Логическая разбивка по дням ("Что логично посмотреть в первый день, а что во второй?"). Приоритизация активностей под интересы. | Сама структура JSON для маршрута. |
| **Поиск мест** | Формирование семантического запроса для RAG ("найди исторические места, похожие на описание 'древний кремль'"). | Векторный поиск (FAISS) по базе знаний. Вызов Google Maps Places API по названию места. |
| **Поиск отелей** | Интерпретация "бюджетный отель рядом с центром" в параметры API (цена, район). | HTTP-запрос к API отелей. Парсинг структурированного JSON-ответа. |
| **Генерация описаний** | Создание дружелюбного, связного текста для каждого дня и достопримечательности на основе фактов (цитат из RAG). | Сами факты (названия, часы работы, координаты) подставляются из данных. |
| **Экспорт в PDF** | Нет. | Сбор данных в шаблон PDF или создание .ics файла для календаря. |
| **Экспорт в Календарь** | Использование Tool calls и Api Google Calendar. | Проверка структуры и формата ответов модели. |
