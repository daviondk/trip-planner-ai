## 2. Политика логов и работа с персональными данными

**Что логируется:**
*   **user_id:** Псевдонимизированный идентификатор (hash от chat_id).
*   **session_id:** Уникальный ID сессии.
*   **timestamps:** Время запросов.
*   **intents:** Классифицированный интент запроса (plan_trip, change_plan, etc.).
*   **tool_calls:** Какие инструменты вызывались (search_hotel, get_map) и успешность их выполнения.
*   **latency и токены:** Для мониторинга производительности и стоимости.
*   **обезличенный текст запроса:** Текст, прошедший через PII-анонимайзер ([Имя] -> [PERSON]).

**Что НЕ логируется:**
*   Сырой (raw) текст пользователя до анонимизации.
*   Реальные имена, адреса, геолокация (точные координаты заменяются на название района).
*   Ответы LLM, содержащие потенциальные PII.

**Политика хранения:**
*   Логи хранятся локально (с определенным TTL). По умолчанию сессионные данные (state) живут только в памяти и удаляются через 1 час неактивности.

## 3. Защиты от Injection, Jailbreak и подтверждение действий

### Многослойная защита
1.  **Input Layer:**
    *   **Анонимизация:** Замена PII на плейсхолдеры.
    *   **Валидация:** Проверка, что ввод пользователя не превышает лимит символов (например, 2000 символов).
    *   **Injection Scanner:** Поиск паттернов "ignore previous instructions", "you are now ChatGPT" и т.д. При обнаружении — запрос помечается как подозрительный, и агент переходит в режим "безопасного ответа" (извиниться и попросить перефразировать).

2.  **Orchestration Layer (LangGraph):**
    *   **Изоляция промптов:** Системный промпт для каждого узла (агента) жестко задан в коде и не подлежит изменениям извне.
    *   **Структурированный вывод:** Там, где это возможно, мы требуем от LLM вывода в формате JSON (например, `{"city": "Moscow", "days": 3}`), который затем парсится кодом. Это снижает риск выполнения нежелательных инструкций.
    *   **Контроль переходов:** Переход в следующий узел (например, от сбора пожеланий к поиску отелей) происходит только после нажатия пользователем кнопки "Подтвердить" или после успешного завершения tool call'а, а не на основе текстового ответа LLM.

3.  **Tool Execution Layer:**
    *   Все вызовы внешних API и доступ к файловой системе осуществляются через строго типизированные функции (tools) с фиксированными параметрами. LLM не имеет возможности выполнять произвольный код.
    *   **Валидация параметров:** Перед вызовом `search_hotel(destination: str, budget: int)` код проверяет, что `budget` — это число, а не "DROP TABLE USERS".

### Подтверждение действий (Human-in-the-Loop)
**Бронирование:** В PoC мы не проводим реальные транзакции. При нажатии кнопки "Забронировать" пользователь получает ссылку на внешний сайт (как максимум) с предзаполненными данными, но оплата происходит там. Мы **не храним** и **не подтверждаем** платежные данные.
